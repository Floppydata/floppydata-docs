<Note>
  **In Floppydata, the username is not just for authentication – it defines how your request is routed.**

  Instead of selecting proxies manually, you pass routing instructions inside the username. These instructions are called **Smart Parameters**.
</Note>

Every request includes these parameters, and Floppydata uses them to determine:

- which IP type to use
- where the request should originate from
- whether the IP should rotate or stay the same
- how traffic should be routed

## The core idea

## How parameters are passed

Smart Parameters are embedded directly in the username.

**Format**

```bash
USERNAME-PARAM1-VALUE1-PARAM2-VALUE2
```

**Example**

```bash
user123-res-country-US-city-New_York-session-abc123
```

**This means:**

> _use residential proxies route through the US narrow to New York keep the same IP for this session_

## Username structure

A username consists of two parts:

- a **base credential**
- a **chain of Smart Parameters**

```bash
user123-res-country-US-city-Chicago-session-s01
```

| Parameter | Example | Description |
| --- | --- | --- |
| Base username | `user123` | Your main account identifier |
| Proxy type | `res` | Defines the proxy network |
| Country | `country-US` | Sets the exit country |
| City | `city-Chicago` | Narrows routing to a city |
| Session | `session-s01` | Keeps the same IP |

<Note>
  Each parameter adds one layer of routing logic.
</Note>

## Smart Parameters reference

Below is a reference for the most common Smart Parameters and how they affect routing.

### `type` –  IP type

Defines which proxy network is used.

| Parameter | Value | Description | Example |
| --- | --- | --- | --- |
| `type` | `res` | Residential | `user123-res` |
| `type` | `mobile` | Mobile (4G/5G) | `user123-mobile` |
| `type` | `dc` | Datacenter | `user123-dc` |
| `type` | `isp` | Static ISP | `user123-isp` |

<Note>
  If omitted, the default is `res`.
</Note>

### `country` –  exit country

Defines where the request should originate from.

| Parameter | Value | Description | Example |
| --- | --- | --- | --- |
| `country` | `US` | United States | `user123-res-country-US` |
| `country` | `DE` | Germany | `user123-res-country-DE` |
| `country` | `GB` | United Kingdom | `user123-res-country-GB` |

<Note>
  Uses ISO country codes. If omitted, global routing is used.
</Note>

### `city`  –  exit city

Provides more precise geo-targeting.

| Parameter | Value | Description | Example |
| --- | --- | --- | --- |
| `city` | `New_York` | New York | `user123-res-country-US-city-New_York` |
| `city` | `Berlin` | Berlin | `user123-res-country-DE-city-Berlin` |

<Note>
  - use underscores instead of spaces
  - not available in all locations
  - falls back to country-level routing if unavailable
</Note>

### `state`  –  US state

Used for US-only targeting.

| Parameter | Value | Description | Example |
| --- | --- | --- | --- |
| `state` | `CA` | California (abbreviation) | `user123-res-country-US-state-CA` |
| `state` | `California` | California (full name) | `user123-res-country-US-state-California` |

### `session` –  sticky IP

Keeps the same IP across requests.

| Parameter | Value | Description | Example |
| --- | --- | --- | --- |
| `session` | `user01` | Keeps the same IP for this session | `user123-res-country-US-session-user01` |

**Behavior**

| Session state | Result |
| --- | --- |
| same session | same IP |
| different session | different IP |
| no session | rotation |

<Note>
  Session lifetime: up to 10 minutes of inactivity.
</Note>

### `rotation` –  force IP change

Overrides session behavior.

| Parameter | Value | Description | Example |
| --- | --- | --- | --- |
| `rotation` | `1` | new IP every request | `user123-res-country-US-rotation-1` |
| `rotation` | `0` | keep IP within session | `user123-res-country-US-rotation-0` |

### `protocol` –  connection type

Usually set via port, not via username.

| Parameter | Value | Description | Example |
| --- | --- | --- | --- |
| `protocol` | `HTTP` | Use HTTP proxy connection | `geo.g-w.info:10080` |
| `protocol` | `SOCKS5` | Use SOCKS5 proxy connection | `geo.g-w.info:10800` |

## How parameters work together

Each parameter narrows down routing.

**Example**

```bash
user123-res-country-US-city-Chicago-session-order01
```

| Aspect | Result |
| --- | --- |
| IP type | Residential |
| Country | United States |
| City | Chicago |
| Session | Same IP for this session |

<Note>
  The more parameters you add, the more specific the routing becomes.
</Note>

## Using Smart Parameters in requests

Smart Parameters are passed as part of the proxy connection.

**Format**

```bash
http://USERNAME:PASSWORD@geo.g-w.info:10080
```

**Example**

```bash
http://user123-res-country-US-session-s01:PASSWORD@geo.g-w.info:10080
```

<Note>
  Every request using this connection follows the same routing rules.
</Note>

### Build username via API

If you prefer not to build the string manually, you can generate it through the API.

```bash
curl "https://client-api.floppy.host/v1/rotating/smartParamsBuilder?type=res&country=US&city=New_York&session=abc123" \
  -H "X-Api-Key: YOUR_API_KEY"
```

<Note>
  Returns a ready-to-use username.
</Note>

### Real examples

| Use case | Example | Result |
| --- | --- | --- |
| Basic rotating traffic | `user123-res-country-US` | residential, US, rotating |
| Sticky session | `user123-res-country-US-session-login01` | same IP, session-based |
| Mobile proxy | `user123-mobile-country-US` | mobile network, high trust |
| City targeting | `user123-res-country-US-city-Chicago` | precise geo targeting |
| Static ISP | `user123-isp-country-US` | fixed residential IP |
| Datacenter rotation | `user123-dc-rotation-1` | fast, always new IP |

## Common mistakes

### Incorrect format

| Wrong | Correct |
| --- | --- |
| `user123-countryUS` | `user123-country-US` |

### Expecting rotation with a session

If you set:

```bash
-session-s01
```

The IP will stay the same.

### Wrong proxy type for the task

| Expectation | Actual configuration | Result |
| --- | --- | --- |
| high trust traffic | datacenter IPs | lower trust, higher risk of blocks |

<Warning>
  Always match the proxy type to your use case.
</Warning>
